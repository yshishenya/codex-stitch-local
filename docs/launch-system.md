# Launch System

This file turns repository promotion into a repeatable execution system.

## Goal

Get the first wave of **real installs, real feedback, and real stars** for `StitchFlow` without fake engagement.

Primary success metrics:

- GitHub stars
- install attempts
- issues / discussions / feature requests
- replies from design engineering and agent-builder communities

## Core message

Short version:

> Prompt -> UI directions -> local HTML + screenshots

Expanded version:

> StitchFlow turns Google Stitch into a portable local workflow for generating UI directions, Tailwind-friendly HTML, and screenshots across Codex, Claude Code, OpenClaw, GitHub Copilot, and Gemini CLI.

## Primary audiences

1. Product and design engineers
   - want fast UI exploration before frontend implementation
   - care about editable local output, not only hosted demos

2. Agent builders
   - want one reusable skill/workflow across multiple clients
   - care about portable packaging and installability

Secondary audiences:

- founders prototyping product directions
- internal tools teams
- AI coding enthusiasts looking for prompt-to-UI workflows

## Positioning rules

Keep repeating the same angles:

- local-first output
- prompt-to-HTML and screenshots
- portable across agent clients
- practical workflow, not just a raw SDK wrapper

Avoid leading with:

- architecture details
- internal file layout
- compatibility trivia
- generic “AI design” wording without concrete outputs

## Asset checklist

Minimum launch assets:

- 1 hero screenshot
- 1 short GIF or video showing prompt -> result
- 3 copy-paste prompts
- 1 README quickstart
- 1 short post per channel
- 1 FAQ block for predictable objections

## Canonical proof points

Use these in posts and replies:

- built on Google Stitch SDK
- generates UI directions, edits, and variants
- saves local HTML and screenshots
- works across Codex, Claude Code, OpenClaw, GitHub Copilot, and Gemini CLI
- one canonical skill slug: `stitchflow`

## 7-day rollout

### Day 1 — packaging

- tighten README first screen
- make quickstart obvious
- add 3 prompt examples
- ensure demo assets are easy to see

### Day 2 — content prep

- prepare Show HN text
- prepare 3 X posts
- prepare 2 Reddit variants
- prepare short Discord / Telegram post

### Day 3 — friendly soft launch

- personal network
- agent communities
- design engineering chats
- ask for install feedback, not vanity stars

### Day 4 — major public post

- Show HN or equivalent
- respond quickly to comments
- collect objections and convert them into README / FAQ improvements

### Day 5 — ecosystem distribution

- catalogs
- marketplace-style lists
- awesome lists
- plugin / extension ecosystems

### Day 6 — technical content

- DEV.to or blog post
- explain why local artifacts matter
- compare raw SDK vs reusable workflow

### Day 7 — iteration

- update README from feedback
- pin best examples
- publish roadmap and good-first-issues

## Posting rules by surface

### Hacker News

Lead with:

- the practical problem solved
- local artifacts
- multi-client support

Do not lead with:

- marketing language
- “revolutionary” framing
- vague AI hype

### Reddit

Adapt per subreddit.

- r/opensource -> open-source workflow and local outputs
- r/webdev -> HTML / Tailwind / prototyping angle
- r/ClaudeAI or coding-agent communities -> cross-agent workflow angle

### X / Twitter

Use 1 strong visual and 3 short bullets.

### Discord / Telegram

Keep it short and conversational.
Ask for feedback or installs, not only stars.

## Objections to pre-answer

### “Why not use the raw Stitch SDK?”

Because most teams need a reusable workflow with install docs, prompts, local artifacts, and cross-client support — not only a low-level SDK.

### “Why local output matters?”

Because teams want HTML and screenshots they can review, diff, share, and build from immediately.

### “Who is it for?”

Product/design engineers and agent builders first.

### “Is this a hosted product?”

No. It is a local workflow / skill bundle around Stitch.
